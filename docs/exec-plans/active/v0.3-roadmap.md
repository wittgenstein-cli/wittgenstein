## Purpose

This plan exists because the repo just received a structured external audit. The audit's diagnosis: **doctrine has run ahead of code by ~1.5 milestones**. The repo's governance (15 ADRs, 5 RFCs, 11 Briefs, 11 issue templates, two explicit decision lanes, automated guardrails) is stronger than the code that implements the thesis (image goldens still empty; first VQ decoder bridge not landed; reproducibility doctrine has no testable golden gate).

This plan converts that diagnosis into 1-2 weeks of executable work, with clear ownership, acceptance gates, and a deliberate bias toward **engineering-lane PRs that make existing doctrine "grow teeth"** rather than new doctrine.

It does **not** open new ADRs except where the audit explicitly called for one (ADR-0016 for the untrusted-code-execution boundary).

## v0.3 release gate (the forcing function)

**v0.3 ships only when all of the following are true:**

1. **At least one byte-for-byte golden** is checked into `fixtures/golden/` and gated by CI. (Sensor is the easiest path — pure NumPy / pure TS, no neural decoder dependency.)
2. **M2 sweep-level kill criterion verified.** Either Kokoro-82M passes CPU-deterministic replay on the parity-tts test (Brief I §H I.7), or Piper takes its place via the named fallback path, or — if both fail — `say` / `espeak-ng` becomes the v0.3 audio path with the kill explicitly logged.
3. **`packages/core` test:src ratio ≥ 0.4** and **`packages/cli` test:src ratio ≥ 0.25.** These are the surfaces the manifest spine actually runs on; current ratios (0.05 / 0.04) leave the reproducibility promise unverified.
4. **README "Receipts" table is evidentiary-only** — every row reproducible by `pnpm` from a clean checkout, or the row is moved to "Demos / Showcase" and labeled as such.
5. **Supply-chain CI is gating, not advisory.** `dependency-review-action` set to `fail-on-severity: high`; CodeQL on `security-extended`; codec-cruiser rule prevents codec-X importing codec-Y.

These five gates are the difference between "doctrine that promises reproducibility" and "code that demonstrates it." The v0.3 release notes should cite the commits that turned each gate from doctrinal claim into checkable fact.

## M2 finish — remaining slices

Sequenced. Each slice is its own PR.

### Slice C2 — Kokoro-82M backend wiring

**Owner-line.** Audio.
**Inputs.** ADR-0015, Brief I §H I.1, current `procedural-audio-runtime` placeholder in `packages/codec-audio`.
**Scope.**

- Add a Kokoro-82M ONNX runtime backend selectable behind a `WITTGENSTEIN_AUDIO_BACKEND=kokoro` env variable (default stays procedural until parity tests certify the switch).
- Pin the Kokoro weights by SHA-256 in a new `packages/codec-audio/decoders/kokoro/manifest.json`. Lazily download on first use; cache under `~/.cache/wittgenstein/decoders/kokoro/`.
- Replace `decoderId = procedural-audio-runtime` with `decoderId = kokoro-82M:<weights-sha256>` when the backend is engaged.
- ONNX runtime CPU execution provider only at this slice; GPU is a follow-up.

**Acceptance gate.**

- `pnpm --filter @wittgenstein/codec-audio test` green.
- A new `tts --backend kokoro --dry-run` smoke produces a manifest with `audioRender.decoderId = kokoro-82M:<hash>` and `determinismClass: structural-parity`.
- Three back-to-back `tts --backend kokoro --seed 7` invocations on the same machine produce byte-identical WAV (this is the determinism probe; if it fails, Slice C2 falls back to Piper and Brief I §H I.2 promotes from 0.3 → 0.6).

**Out of scope.**

- GPU backend.
- Voice cloning.
- New routes.

**Tracked as.** Issue (TBD — Codex creates on slice start).

### Slice C3 — Parity tests for all three audio routes

**Owner-line.** Audio.
**Inputs.** Brief J §Verdict 4 (fixture-class decision tree), Slice C1's `audioRender` schema, Slice C2's deterministic-replay verification.
**Scope.**

- `packages/codec-audio/test/parity-tts.test.ts` — for `kokoro` backend: structural parity (sampleRate exact, channels exact, duration ±5%, UTMOS within band). For `procedural` backend: byte-parity SHA-256 against `fixtures/golden/audio/tts/`.
- `packages/codec-audio/test/parity-soundscape.test.ts` — byte-parity SHA-256 against `fixtures/golden/audio/soundscape/`.
- `packages/codec-audio/test/parity-music.test.ts` — byte-parity SHA-256 against `fixtures/golden/audio/music/`.
- `packages/codec-audio/test/route-collapse.test.ts` — assert no shared-mechanical scaffolding > 30 lines duplicated across sibling routes (Brief J §Kill criterion 1).
- Goldens checked into `fixtures/golden/audio/{tts,soundscape,music}/`. Initial fixtures use the procedural backend for tts (so the structural parity test can be authored before Kokoro is fully stable).

**Acceptance gate.**

- All four tests green on `pnpm --filter @wittgenstein/codec-audio test`.
- CI matrix runs both `WITTGENSTEIN_AUDIO_BACKEND=procedural` and `WITTGENSTEIN_AUDIO_BACKEND=kokoro` (the latter may be a CI-skipped job if weights aren't available in CI).
- One row in README "Receipts" reads "audio parity: 3 routes × byte-or-structural fixtures (CI-gated)".

**Out of scope.**

- Adding new routes.
- Replacing existing fixtures with neural-generated ones.

### Slice E — Sweep-level kill criterion verification

**Owner-line.** Audio + research.
**Inputs.** Brief I §Kill criteria §1.
**Scope.**

- Run the parity-tts test with `kokoro` backend on a clean macOS + Linux + Windows matrix (or at least macOS + Linux Docker).
- If Kokoro is byte-deterministic on CPU on all platforms tested: Brief I status flips to `🟢 Verified v0.2`; ADR-0015 needs no amendment.
- If Kokoro is non-deterministic on CPU even with `torch.use_deterministic_algorithms(True)` / ONNX deterministic flags: open ADR-0015 amendment (or supersede with ADR-0017) naming Piper as the v0.3 default; Slice C2 + C3 redo against Piper.
- If both fail: open ADR-0017 ratifying procedural-only TTS via `say`/`espeak-ng` for v0.3; Brief I `## Verdict §1` flips to fallback path; release notes log the kill verbatim.

**Acceptance gate.**

- Brief I `## Verdict §1` either confirmed or amended with a paired ADR.
- Slice C2 `decoderId` correctly reflects the verified backend.

## P1 backlog — within 2 weeks

### P1.1 — `core` and `cli` test ratio raise

`packages/core` test:src is 0.05 (78 / 1,575); `packages/cli` is 0.04 (23 / 632). These are the surfaces that implement the manifest-spine reproducibility promise. The audit's red line:

- `packages/core` ≥ **0.4** (priority files: `runtime/manifest.ts`, `runtime/seed.ts`, `runtime/router.ts`, `runtime/budget.ts`, `runtime/retry.ts`).
- `packages/cli` ≥ **0.25** (priority files: each `commands/*.ts` entrypoint, config resolution, `--manifest` replay path).

The ratio threshold itself should land in `docs/engineering-discipline.md` as a `pre-merge check` for new code in those two packages. (This is doctrine — needs a small ADR.)

**Tracked as.** Issue (TBD) + paired ADR-0017 (test-ratio doctrine).

### P1.2 — Supply-chain CI hardening

- `dependency-review-action`: change `continue-on-error: true` → `fail-on-severity: high` and `deny-licenses: GPL-3.0, AGPL-3.0` (Apache-2.0 incompatibility).
- CodeQL: language matrix, switch query suite from `default` to `security-extended`; for Python add explicit `setup-python` + `pip install -r polyglot-mini/requirements.txt` so source paths are visible.
- Add a `pull_request_target` lock-down comment to `doctrine-guardrail.yml`: "NEVER add `actions/checkout` with `ref: ${{ github.event.pull_request.head.ref }}` to this workflow." Encode the rule in `engineering-discipline.md`.

**Tracked as.** Issue (TBD).

### P1.3 — Codec-independence guardrail

The hard-constraint "codecs do not depend on each other; codecs depend on `schemas`" is currently honored by convention. Add a `dependency-cruiser` (or `madge`) rule that fails CI if `packages/codec-X/**` imports `packages/codec-Y/**`.

**Deliverable.**

- `.dependency-cruiser.cjs` with one rule.
- `pnpm lint:deps` script invoked from the existing lint job.

**Tracked as.** Issue (TBD).

### P1.4 — VQ decoder bridge spike (image M1B preview)

The audit's most consequential point: ROADMAP Phase 1 has 5 unchecked boxes for "image neural codec real" and they have not budged in 2 weeks. The first VQ decoder bridge — even a deliberately-narrow LlamaGen wrapper that only handles a fixed prompt — flips `decoders/llamagen.ts` from `throw new Error("NotImplemented")` to "produces real latents → PNG." That single PR converts the Phase 1 ⚠️ to ✅ for the first time.

**Scope.**

- `packages/codec-image/decoders/llamagen.ts` (or whichever LFQ-family decoder ADR-0005 + Brief A endorse) accepts a fixed scene-spec, returns real PNG bytes.
- One golden image checked into `fixtures/golden/image/`.
- `decoderId = llamagen-7B:<weights-sha256>` written to manifest.

**Tracked as.** Issue (TBD) — sibling to existing `Issue #70 — M1B handoff`.

### P2.3 — Brief lineage DAG (`docs/research/briefs/lineage.mermaid`)

11 briefs (A–J, plus the audit memo) interlock in non-obvious ways (C revised; A refreshed; I → ADR-0015; J → exec-plan). A mermaid DAG of brief / RFC / ADR / exec-plan relations (the kind OLMo and PEP processes maintain) makes the chain navigable.

**Tracked as.** Issue (TBD).

### P2.4 — ADR-0016 untrusted-code-execution boundary

`polyglot-mini` runs LLM-generated painting code under `subprocess + 20s timeout + safe globals`. README and `SECURITY.md` are honest that this is research-grade. The audit recommends ratifying the current research-grade-vs-production-grade boundary as ADR-0016, and naming the future production path (`nsjail` / `bubblewrap` / Pyodide-WASM) explicitly.

**Tracked as.** Issue (TBD) + paired ADR-0016 PR.

### P2.5 — Doctrine freeze sprint

ADR-0014 §Kill criterion already says "if the governance lane produces > 1 ADR per week sustained for 4 weeks, pause and reconsider." The audit shows recent PR cadence has skewed governance-heavy. The forcing function: the next 3 weeks open **zero** new ADRs except those explicitly named here (0016, 0017). Engineering-lane PRs only.

**Tracked as.** Issue (TBD) — a single tracking issue, not a PR.

## Risks

- **Kokoro non-determinism kills the v0.3 audio thesis.** Mitigation: Brief I already has a fallback (Piper) and a kill-fallback (`say`/`espeak-ng`); ADR-0017 is the planned response.
- **Test-ratio raise (P1.1) is the heaviest item and may slip past Week 2.** Mitigation: write the threshold ADR first; let it land independently of the actual test-writing; Codex-grade test scaffolding can run in parallel with Slice C2/C3.
- **Doctrine freeze (P2.5) is socially hard to enforce.** Mitigation: the doctrine-guardrail workflow already nudges; ADR-0014's existing kill criterion is the formal escape hatch.
- **VQ decoder bridge (P1.4) is the most upside-volatile item.** It depends on a usable LFQ-family weights release. Mitigation: even a narrow proof-of-concept with a fixed prompt converts the README ⚠️ to ✅ for the first time; full M1B is downstream.

## The three open questions

### G1. Discrete-token vs diffusion decoder

**Question:** is the first image decoder we integrate a discrete-token decoder (LFQ / FSQ family, per Brief A) or a diffusion decoder (SDXL-class) with a constrained prompt path?

**Pointers to investigate:**

- **LFQ / FSQ / MAGVIT-v2 (Google 2024, arXiv:2310.05737)** — Brief A's preferred vocabulary. Tokenizer is explicit, decoder is a transformer, next-token prediction handles the image. Matches ADR-0005 (frozen-decoder) cleanly.
- **Emu3 (BAAI 2024, arXiv:2409.18869)** — pure NTP over a unified tokenizer. If the image path is just "LLM emits image tokens, decoder decodes," the Codec<SceneSpec, Image> shape compresses further. Threat to the "scene-spec JSON" layer: does it still earn its keep?
- **GPT-4o native image generation (OpenAI 2025)** — partial-redraw / edit-in-place behavior suggests token-level interleaving of text and image tokens inside the base model. Closed model; reasoning from reports only. Useful as a capability target, not as an implementation reference.
- **Diffusion path (SDXL / SD3 / Flux)** — rejected by ADR-0005 for the *core* image path, but the rejection is "no diffusion inside the default decoder," not "no diffusion ever." If a `codec-image-sdxl` ships as an opt-in non-default adapter, that is consistent with the hard constraint. Worth naming explicitly so it is not reopened by accident.

**What to produce in the follow-up:** a two-column table — *discrete-token candidate* vs *diffusion candidate* — with integration cost, licensing, frozen-hash stability, and the ADR-0005 compatibility read.

### G2. Training data source for the adapter (L4)

**Question:** when `codec-image` grows a real L4 adapter (scene-spec → latent bridge), what data does it train on?

**Pointers to investigate:**

- **Public caption-image pairs (LAION-5B, CC12M, DataComp-1B)** — widely used, licensing is messy, quality is lumpy. Baseline option.
- **Synthetic scene-spec pairs generated by a strong captioner + existing artifact library** — our showcase has 35 artifacts with matching SceneSpec JSON; at 1000× synthetic expansion it is training-data-sized but will inherit any bias in the captioner.
- **Self-play (`produce` → re-captioner → produce')** — cheap, in-repo, closes the loop but risks mode collapse. Known failure mode from distillation literature; cite CycleGAN-style pitfalls.
- **Licensed data + frozen decoder** — the "boring safe" option: pay for a clean dataset, freeze the decoder weights as the artifact hash, make no claim that data or decoder are novel.

**What to produce in the follow-up:** a decision between these four with a concrete row in `docs/hard-constraints.md` about what training data is in-bounds and what is not.

### G3. Packaging form (MCP / npx / skill / curl)

**Question:** how does `codec-image` ship to users who are not cloning the repo — and is it an MCP server, an `npx` one-liner, a Claude Skill, or a `curl | sh` wrapper?

**Pointers to investigate:**

- **MCP (Model Context Protocol, Anthropic 2024–2026)** — the emerging default for exposing tools to Claude / Cursor. One `codec-image` MCP server, one tool (`produce_image`), schema-driven. Natural fit for Codec v2 because the tool surface *is* the `Codec<Req, Art>` signature.
- **`npx @wittgenstein/codec-image`** — matches the CLI ergonomics locked in RFC-0002. Zero-install one-liner; falls through to the same binary the repo ships. Lowest friction for a curl-age user.
- **Claude Skills / agent primers** — the AGENTS.md-style documentation shape; not a distribution channel by itself, but pairs with whichever binary form ships.
- **`curl | sh` install** — matches the `gh` / `rustup` convention. Highest trust cost (users must trust the domain) but lowest friction for reproducible CI.

**Constraint from the user:** packaging form is *not urgent*. Do not block M1 on this; the brief captures the question so it does not get forgotten. The decision lands after the CLI surface stabilizes post-RFC-0002.

**What to produce in the follow-up:** a packaging matrix with one row per form and a ranked "ship this first" recommendation keyed to the v0.2 → v0.3 → v0.4 release train.
